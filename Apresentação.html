🎯 2. Princípios SOLID Aplicados

S - Single Responsibility

"Uma classe deve ter um, e somente um, motivo para mudar"

// ✅ Cada classe tem UMA responsabilidade
class ProdutoDados { // Só acessa dados }
class ProdutoServico { // Só regras de negócio }
class TelaPDV { // Só interface gráfica }

O - Open/Closed

"Aberto para extensão, fechado para modificação"

// ✅ Interface ABERTA para extensão
interface IFormaPagamento { }
class PagamentoNovo implements IFormaPagamento { }

L - Liskov Substitution

"Subtipos devem ser substituíveis por seus tipos base"

// ✅ Qualquer IFormaPagamento funciona
void finalizar(IFormaPagamento p) {
    p.processarPagamento(total);
}

I - Interface Segregation

"Interfaces menores e específicas são melhores"

// ✅ Interfaces específicas
interface IProdutoDados { }  // Só produto
interface IVendaDados { }    // Só venda

D - Dependency Inversion

"Dependa de abstrações, não de implementações"

// ✅ Depende da interface
class ProdutoServico {
    private final IProdutoDados dados;
}

📚 3. GRASP

🎯 Especialista na Informação

Quem tem a informação processa.

class VendaItem {
    double getSubtotal() {
        return preco * quantidade;
    }
}

🔗 Baixo Acoplamento

Depender de interfaces, não implementações.

class TelaPDV {
    private final ProdutoServico servico;
}

📦 Alta Coesão

Responsabilidades fortemente relacionadas.

class ProdutoDados {
    buscarPorCodigo()
    buscarPorId()
    atualizarEstoque()
}

🏭 Criador

Quem tem a informação cria os objetos.

NotaFiscalBuilder.getNota(tipo);

🎮 Controlador

Delega para um controlador (não para UI).

produtoServico.buscarPorCodigo(codigo);

🪟 5. JFrame e Componentes Swing

Thread que processa eventos da interface.

⚠️ Evitar setLayout(null)

🔄 Console vs GUI

📟 Console (Sequencial)

Executa linha por linha. Trava esperando entrada.

System.out.print("Digite: ");
String codigo = scanner.nextLine();

🖱️ GUI (Eventos)

Fica em "escuta". Executa quando evento ocorre.

botao.addActionListener(e -> {
    String codigo = campo.getText();
});

🚫 6. Classes Deus e Como Evitá-las

❌ Classe Deus (Antipadrão)

Uma classe que faz TUDO.

class ClasseDeus {
    // Conexão com banco
    // CRUD de produtos
    // Regras de negócio
    // Interface gráfica
    // TUDO junto!
}

✅ SRP (Solução)

Cada responsabilidade em uma classe.

ConexaoMySQL    // Conexão
ProdutoDados    // Acesso
ProdutoServico  // Regras
TelaPDV         // UI

🔀 7. Separação UI vs Lógica de Negócio

🎨 UI (TelaPDV)

Cria componentes, captura eventos.
NÃO contém regras de negócio

produtoServico.buscar(codigo);

⚙️ Serviço

Validações, cálculos, regras.
NÃO conhece a UI

if(estoque >= qtd) {
    // processar
}

🗄️ Repository

SQL e operações no banco.
NÃO sabe de UI nem regras

"SELECT * FROM produtos"

🔄 8. Inversão de Dependência

// ✅ Dependendo de abstrações
class ProdutoServico {
    private final IProdutoDados dados;  // Interface!
    
    ProdutoServico(IProdutoDados dados) {
        this.dados = dados;
    }
}

// ✅ Injeção no Main
IProdutoDados dados = new ProdutoDados(conexao);
ProdutoServico servico = new ProdutoServico(dados);

🧪 Testabilidade

Podemos criar Mocks para testar sem banco real.

IProdutoDados mock = new MockDados();
ProdutoServico servico = new ProdutoServico(mock);

🔌 Modularidade

Trocar implementação sem afetar o sistema.

// Trocar MySQL por PostgreSQL
IConexao conexao = new ConexaoPostgreSQL();
