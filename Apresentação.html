🎯 Princípios SOLID Aplicados

// ✅ S - Single Responsibility (Responsabilidade Única)
public class ProdutoDados {      // Só lida com acesso a dados
    public Optional<Produto> buscarPorCodigo(String codigo) {...}
    public boolean atualizarEstoque(int id, int qtd) {...}
}

public class ProdutoServico {    // Só lida com regras de negócio
    public Optional<Produto> buscarPorCodigo(String codigo) {...}
    public boolean temEstoque(Produto p, int qtd) {...}
}
// ✅ O - Open/Closed (Aberto/Fechado)
public interface IFormaPagamento {
    String getNome();
    boolean processarPagamento(double total, double pago);
}

// Para adicionar nova forma de pagamento:
// 1. Criar nova classe implementando IFormaPagamento
// 2. NENHUM código existente precisa ser modificado!
// ✅ L - Liskov Substitution (Substituição)
public void finalizarVenda(IFormaPagamento pagamento) {
    // Funciona com Dinheiro, Cartão, PIX, QUALQUER UM!
    pagamento.processarPagamento(total, valorPago);
}
// ✅ I - Interface Segregation (Segregação)
public interface IProdutoDados {    // Só métodos de PRODUTO
    Optional<Produto> buscarPorCodigo(String codigo);
    boolean atualizarEstoque(int id, int qtd);
}

public interface IVendaDados {       // Só métodos de VENDA
    int salvarVenda(Venda venda);
    void salvarItens(int vendaId, List<VendaItem> itens);
}
// ✅ D - Dependency Inversion (Inversão de Dependência)
public class ProdutoServico {
    private final IProdutoDados dados;  // ← INTERFACE!
    
    public ProdutoServico(IProdutoDados dados) {
        this.dados = dados;  // Injeção de dependência
    }
}

🏛️ Arquitetura em Camadas

🎨 CAMADA DE APRESENTAÇÃO (tela/TelaPDV)
Interface gráfica (JFrame, JTable, JButton) | Eventos do usuário | NÃO contém regras de negócio
⚙️ CAMADA DE SERVIÇOS (servico/ProdutoServico)
Regras de negócio | Orquestração entre repositórios | Validações
🗄️ CAMADA DE REPOSITÓRIOS (dados/ProdutoDados)
Acesso a dados (SQL) | CRUD no banco | NÃO contém regras de negócio
🔌 CAMADA DE CONEXÃO (banco/ConexaoMySQL)
Conexão com MySQL | Gerenciamento de transações

🔄 Fluxo de uma Venda (Com as Camadas)

👤 Usuário clica "Buscar"
→
🎨 TelaPDV.realizarBusca()
→
⚙️ ProdutoServico.buscarPorNome()
→
🗄️ ProdutoDados.buscarPorNome()
→
🔌 ConexaoMySQL.conectar()
👤 Usuário clica "Finalizar"
→
🎨 TelaPDV.abrirDialogPagamento()
→
✅ DialogPagamento.confirmarPagamento()
→
⚙️ VendaServico.finalizar()
→
🏭 NotaFiscalBuilder.getNota()

📊 Benefícios da Arquitetura

Benefício Como é alcançado
🧩 ManutenibilidadeCódigo organizado por responsabilidades
🧪 TestabilidadeCada camada pode ser testada isoladamente
♻️ ReusabilidadeServiços podem ser reutilizados por diferentes UIs
🔄 FlexibilidadeTrocar MySQL por PostgreSQL = mudar 1 classe
📈 EscalabilidadeAdicionar nova forma de pagamento = criar 1 classe
🔗 Baixo AcoplamentoClasses dependem de interfaces, não de implementações
🎯 Alta CoesãoCada classe tem uma responsabilidade clara
